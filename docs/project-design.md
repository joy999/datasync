# DataSync 项目设计说明

## 1. 项目定位

DataSync 是一个分布式数据复制框架，它将系统职责拆成两部分：

1. 组内一致性
2. 组间同步

它的核心设计思想是：这两件事不应该被强行合并成一个全局共识系统。

在单个 `NodeGroup` 内，节点通过 Raft 保持副本状态一致。

在不同 `NodeGroup` 之间，数据通过显式的同步链路进行复制，而不是把所有组都拉进同一个 Raft 集群。

因此，这个项目适合以下场景：

- 多地域部署，每个地域都需要自己的高可用集群
- 生产、容灾、边缘节点之间的数据同步
- 按业务控制特定数据类型在组间复制
- 存储方式和序列化方式由应用自行扩展的同步框架

## 2. 设计目标

从当前仓库结构来看，项目主要追求以下目标。

### 2.1 组内强一致

每个 `NodeGroup` 都应表现为一个小型复制集群。

- 由一个 Leader 接受写入
- 写入转化为 Raft 日志
- 已提交日志被应用到本地存储驱动
- Follower 最终收敛到与 Leader 一致的状态

### 2.2 组间可配置同步

不同组之间应通过显式关系来交换数据。

- Outbound 表示一个组向外推送数据
- Inbound 表示一个组接收其他组的数据
- 同步关系可以是单向或双向
- 组间同步只能由源组 Leader 发起

### 2.3 存储抽象

框架本身不定义业务表结构，也不约束业务模型。

应用数据统一抽象成 `DataRecord`，实际存储由 `StorageDriver` 负责。

这意味着：

- 框架同步的是记录，而不是业务对象
- 驱动决定记录如何存储
- 驱动决定记录如何编码和解码
- 冲突语义留给驱动或业务层处理

### 2.4 统一传输协议

所有跨节点通信都应通过同一种协议外壳承载。

当前设计基于：

- `Envelope` 作为统一消息封装
- Protobuf 作为线协议格式
- 信封中携带来源身份和可选认证信息
- 在同一协议体系内承载不同类型的负载

这样一来，同步消息、Raft 消息、心跳、发现、认证都能复用同一套传输模型。

### 2.5 基础设施可插拔

整个框架围绕可替换层来设计。

- 存储通过 `StorageDriver` 抽象
- 传输通过 `Transport` 抽象
- 认证通过 `AuthManager` 抽象

仓库中当前提供的是默认实现，但整体架构更像适配层，而不是唯一依赖。

## 3. 核心领域模型

### 3.1 Node

`Node` 是系统中的顶层运行单元。

它负责协调：

- Raft 状态复制
- 组元数据
- 同步管理器
- 已注册的存储驱动
- 可选的认证管理器
- 传输层相关的本地节点身份接线

`pkg/node` 是整个系统的组装层。

### 3.2 NodeGroup

`NodeGroup` 是系统中最重要的一致性边界。

它有几个关键属性：

- 同组节点通过 Raft 复制数据
- 一个组可以同步到其他组，也可以接收其他组的数据
- 组是出站和入站同步策略的配置单位
- 只有组 Leader 才允许触发跨组同步

这意味着，一个组同时承担两种边界角色：

- 共识边界
- 复制策略边界

### 3.3 DataRecord

`DataRecord` 是整个项目的通用数据单元。

它包含：

- 记录 id
- 记录类型
- 版本号
- 时间戳
- 负载
- 元数据

框架只把它当成通用复制数据，不解释其中业务语义。

### 3.4 StorageDriver

`StorageDriver` 是最重要的扩展点之一。

它负责：

- 为全量同步读取记录
- 为增量同步读取变更
- 应用收到的记录
- 获取当前版本
- 负责编码和解码记录负载
- 将本地数据变更回传给框架

因此，驱动同时承担三类职责：

- 持久化适配器
- 序列化适配器
- 本地变更源

### 3.5 Transport

`Transport` 抽象了组间消息传递和节点发现能力。

当前实现基于 NATS，并可选接入 JetStream。

它承担的职责包括：

- 按组发布消息
- 订阅组主题
- 发现节点
- 承载同步、Raft、心跳等消息

### 3.6 AuthManager

`AuthManager` 提供跨节点通信的认证边界。

当前实现支持基于令牌的认证与令牌生成，协议层也预留了在 `Envelope` 中携带认证信息的能力。

## 4. 分层架构

整个项目可以理解成五层结构。

### 4.1 应用与驱动层

这是业务数据真正落地的地方。

- 业务数据被包装成 `DataRecord`
- 驱动负责持久化与查询
- 驱动在本地数据变化时触发回调

### 4.2 本地复制层

这是单组内部的 Raft 层。

- 本地写入被提议给 Raft
- Raft 提交复制后的日志
- 已提交记录被解码并应用到存储

### 4.3 组间同步层

这是同步管理器所在的层。

- 源组 Leader 发起同步请求
- 目标侧从驱动读取数据
- 目标侧把记录作为同步消息回传
- 接收侧通过驱动应用这些记录

### 4.4 协议层

这一层对应统一的消息信封。

- 在这里补充消息元信息
- 在这里统一不同负载类型
- 在这里携带认证令牌
- 在这里携带节点与组身份

### 4.5 网络层

这一层对应真实的消息总线。

- NATS 主题路由
- 可选 JetStream 持久化语义
- 节点发现广播

## 5. 关键数据流

### 5.1 本地写入流程

设计上的理想流程是：

1. 应用通过驱动写入数据
2. 驱动触发变更回调
3. Node 收到 `OnDataChange`
4. Node 使用该驱动对应的 `DriverID` 对记录编码
5. 记录被提议到本地 Raft
6. Raft 提交该日志
7. 已提交记录再通过匹配的驱动应用到存储

这样，一个组内就形成了单一的数据状态演进通道。

### 5.2 全量同步流程

设计上的全量同步流程是：

1. 源组 Leader 决定向目标组发起同步
2. 同步管理器发送 `SyncRequest(FULL)`
3. 接收侧从匹配驱动读取记录
4. 接收侧分页发送 `SyncMessage(FULL)`
5. 发送方通过驱动应用这些记录
6. 接收侧发送最终 `SyncResponse`
7. 发送方更新同步状态

### 5.3 增量同步流程

设计上的增量同步流程是：

1. 同步请求带上 `since`
2. 驱动返回该时间点之后的变更
3. 变更记录作为增量消息传输
4. 接收侧应用这些记录
5. 同步状态推进

从 PostgreSQL 驱动的实现来看，这部分原本预期会由类似 CDC 的机制支撑。

### 5.4 Raft 消息转发流程

项目还尝试通过统一传输层转发 Raft 内部消息。

设计流程如下：

1. etcd/raft 产生内部 Raft 消息
2. 本地 Raft 包装层将其编码为协议 `RaftMessage`
3. 传输层把消息发送到组主题
4. 其他节点解码信封，并将消息投递给 `raft.Step`

这样即便 Raft 内部有自己的消息结构，外部仍然可以统一走同一套传输体系。

### 5.5 发现与心跳流程

节点发现和心跳更多提供的是可见性，而不是状态复制。

- 发现机制广播节点身份和领导状态
- 心跳广播节点活跃状态和处理计数
- 本地维护已知节点和已知组
- 同步状态可以借助心跳做基本健康判断

## 6. 为什么要坚持组级共识

这是这个仓库里最关键的设计选择之一。

这个项目并不是要做：

- 把所有节点都放进一个超大 Raft 集群
- 为所有组建立全局有序写入
- 为每一次跨地域写入都建立全局共识

它真正要做的是：

- 在每个业务组内维持强一致
- 在不同组之间做异步或策略驱动的数据复制

这个选择带来的实际好处是：

- 协调成本更低
- 故障域更清晰
- 区域或业务域自治更容易
- 选择性同步更自然
- 在弱网络环境中更现实

## 7. 扩展模型

从当前结构看，仓库主要预留了三类扩展方向。

### 7.1 新增存储驱动

新驱动可以定义：

- 底层存储后端
- 序列化格式
- 变更捕获方式
- 版本策略

### 7.2 替换或扩展传输层

只要保持 `Transport` 接口一致，就可以更换底层网络承载。

### 7.3 增强同步策略

当前同步模型未来可以继续演化为：

- 按数据类型过滤同步
- 基于 checkpoint 的断点续传
- 冲突策略钩子
- 更丰富的拓扑规则

从现有类型定义和 proto 结构看，这个方向是被预留出来的。

## 8. 当前实现状态

目前的实现已经具备框架骨架和关键链路，但并不是所有设计目标都已完全闭环。

### 8.1 已经具备的部分

- 共享核心类型与接口
- 基于 Raft 的本地复制结构
- 基于 NATS 的传输实现
- 统一的 Protobuf 协议
- 支持全量和增量消息交互的同步管理器
- 内存驱动和 PostgreSQL 驱动
- 基于 JWT 的认证管理器
- 基于 `DriverID` 的驱动编码注册机制

### 8.2 最近已补齐的链路

当前代码已经补上了以下几个之前缺失的连接点：

- Raft 消息不再停留在 TODO，而是能通过 transport 转发
- 节点发现现在会广播真实的本地节点身份
- SyncManager 在配置认证后会校验入站消息
- 同步请求 ID 会和响应完成状态关联起来
- 心跳会广播到已知组
- 数据提议阶段会按记录所属驱动使用对应 `DriverID`，而不是只用一个全局驱动 ID

### 8.3 仍然偏简化或未完全闭环的部分

有些设计能力虽然已经出现，但还没有完全做成系统级闭环：

- 发现到的节点还不会自动转成 Raft 成员变更
- GroupManager 目前还是内存态，没有持久化组拓扑和同步策略
- 同步策略能力还没有完全展开到 proto 预留的丰富程度
- 数据冲突解决仍由驱动或业务层负责
- 心跳和状态管理目前还是基础版，不是完整运维视图
- PostgreSQL 的 checkpoint 与 CDC 状态能力已经有模块级实现，但系统级编排仍较有限

## 9. 从设计上推导出的非目标

从当前结构来看，这个项目并不打算成为：

- 完整数据库产品
- 业务表结构定义工具
- 自动解决业务语义冲突的系统
- 对所有组做全局串行写入的系统
- 强绑定某一种存储引擎或某一种编码格式的框架

## 10. 简要总结

一句话概括：

DataSync 试图成为一个通用分布式复制框架，在组内通过 Raft 保证强一致，在组间通过可配置同步链路完成数据交换，并通过统一协议和可插拔基础设施承载整个过程。

如果再压缩成两句短语，就是：

- 组内强一致
- 组间可控同步

这也是我理解里，这个仓库所有设计点真正围绕的中心。
